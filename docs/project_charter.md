# プロジェクト憲章 — カタガミ（目的・学習計画）

| 項目 | 内容 |
|---|---|
| プロジェクト名 | カタガミ（KATAGAMI） |
| 種別 | 個人開発・学習用Webサービス |
| ドキュメント版 | v0.4 |
| 関連文書 | 要件定義書（`requirements_spec.md`） |

> この文書は「**なぜやるか／何を学ぶか**」を扱う。「システムが何を満たすべきか」は要件定義書に分離した。プロジェクトの方向性・学習目標・進め方の意思決定はここに記録する。

---

## 1. 背景

洋裁・ハンドメイドでは「採寸した数値からサイズの合う型紙がほしい」という需要がある一方、既存の本格的な製図ソフトは高機能でも習得コストが高く、「数値を入れるだけ・登録不要・A4で実寸印刷」という最も手前の層は手薄である。まず自分が作りたいもの（人のTシャツ、子供用の小袋、手提げ、犬服など）を実用できる道具として作り、それを公開・運営する一連の流れを学習として経験する。

## 2. プロジェクトの目的

1. **道具として**：自分が作りたい型紙を、自分で実用できる精度で出力する。
2. **学習として**：その制作物をWebサービスとして公開し、運営・計測・改善・収益化の流れを一通り体験する。

> 位置づけは「半々」。道具としての品質（実寸精度）と、公開サービスとしての品質（見つけやすさ・ポリシー・問い合わせ）の両方を大切にする。

## 3. 学習目標（このプロジェクトで身につけたいこと）

> これらは「目的」であり、システム要件ではない。要件は要件定義書に記載する。

- バージョン管理（Git / GitHub）と、変更履歴を残す習慣
- 静的サイトの公開（ホスティング、独自ドメイン、HTTPS）
- アクセス解析の導入と、数字に基づく改善
- プライバシーポリシー等、サービス運営に必要な体裁
- 収益化（広告・アフィリエイト等）の仕組みと前提条件

## 4. 確定した方針（意思決定ログ）

| 日付 | 論点 | 決定 | 補足 |
|---|---|---|---|
| v0.1 | サイトの位置づけ | 半々（自分用の道具＋運営の学習） | 道具品質とサービス品質の両立 |
| v0.1 | 収益化の方向 | 広告（Google AdSense 等）を目標 | 独自ドメインと記事コンテンツが前提 |
| v0.1 | ホスティング（公開） | サイト公開は別途決定（公開フェーズで選定） | GitHubとは切り離す。候補：Cloudflare Pages / Netlify / 公開用の別リポジトリ等 |
| v0.3 | GitHubの用途 | **非公開リポジトリでドキュメント・進行管理に使う**（サイト公開には使わない） | 無料プランでは非公開リポジトリのPages公開は不可のため、公開は別手段とする |
| v0.2 | 印刷の対応端末 | PC・スマホ両対応（実寸は50mm校正枠で担保） | スマホは倍率UIが弱いため校正枠で吸収 |
| v0.2 | ライセンス | コードMIT／型紙は個人利用自由・商用は出所表記 | コードと出力物で条件を分ける |

> 未確定：アクセス解析ツール、サイト名・ドメイン候補、記事の初回テーマ。要件定義の議論の中で順次確定する。

## 5. 学習ロードマップ（フェーズと完了条件）

各フェーズの完了条件（DoD）を満たしたら次へ進む。広告収益化（フェーズ6）は、独自ドメインと記事コンテンツ（フェーズ5）の整備が前提。

| フェーズ | 内容 | 完了条件（DoD） |
|---|---|---|
| 0 自分用に使える | 7型が生成・実寸印刷できる（達成済み） | 実機で1枚印刷し、50mm枠が実測50mmになる |
| 1 Git管理（非公開） | 非公開リポジトリでコード・ドキュメントを版管理し、Issues等で進行管理 | 履歴が複数コミットあり、非公開リポジトリにpush済み |
| 2 公開 | 公開用ホスティングにデプロイ（GitHubとは別手段。Cloudflare Pages等） | 公開URLで型紙生成・印刷ができる |
| 3 計測 | アクセス解析を導入。最低PV/UU、できれば印刷イベント取得 | ダッシュボードで自分以外のアクセスを確認できる |
| 4 運営の体裁 | プライバシーポリシー・問い合わせ手段を用意。改善を1サイクル回す | ポリシー掲示済み、解析データを根拠に1改善を実施 |
| 5 コンテンツ拡充 | 作り方の読み物を数本。独自ドメイン取得 | 独自ドメインで運用、記事からツールへの回遊がある |
| 6 収益化 | 広告またはアフィリエイト等を導入 | 収益化の仕組みが1つ稼働（金額の多寡は問わない） |

## 6. KPIと改善の進め方

数字を見て改善するための指標。ゴール指標は「印刷実行数」（実質のコンバージョン）に置く。

| 指標 | 意味 |
|---|---|
| PV / UU | ページ閲覧数・訪問者数 |
| 型紙生成数 | パターンがどれだけ使われたか |
| 印刷実行数 | 最重要のコンバージョン |
| 型別の人気 | どのモード・型が使われるか |
| 流入元 | どこから来たか |
| 再訪率 | リピート利用 |

> 「これらを計測できる仕組みを備える」ことは要件であり、要件定義書の非機能要件（計測）に記載する。

## 7. プロジェクト上のリスクと対応

| リスク | 影響 | 対応 |
|---|---|---|
| 公開しても閲覧されない | 学習の手応えが薄い | 検索される具体語でのページ作り、SNS・手芸コミュニティへ共有 |
| 広告審査に通らない | 収益化が進まない | コンテンツ拡充・独自ドメイン・ポリシー整備を先行 |
| 個人開発の継続が止まる | プロジェクト頓挫 | 小さく区切ったフェーズで達成感を積む |

> 製図の不備・印刷倍率ミスといった「システム品質のリスク」は要件定義書に記載する。
